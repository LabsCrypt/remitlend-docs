# Data Model & Postgres Schema

This document describes the PostgreSQL data model for Remitlend, including the core tables used by the indexer, scoring engine, webhook subsystem, and audit trail.

## Why this matters

Contributors need a clear reference for how on-chain Soroban contract events map to the off-chain schema. Without this document, the only authoritative source is the database migrations, which makes onboarding slower and increases the chance of incorrect assumptions.

## Core tables

### `contract_events`
Purpose: store the raw decoded events received from the indexer.

Key columns:
- `event_id` — primary key for the event row.
- `ledger` — Soroban ledger sequence containing the event.
- `transaction_hash` — transaction that emitted the event.
- `contract_id` — contract that produced the event.
- `event_type` — decoded event name (e.g. `LoanRequested`, `LoanRepaid`).
- `topics` — decoded event topics, typically including borrower address and loan ID.
- `data` — decoded event payload values.
- `cursor` — returned by the Stellar RPC to track progress.
- `created_at` — insertion timestamp.

Why it exists: provides an immutable audit log of the exact contract events received from the blockchain.

### `loan_events`
Purpose: store normalized loan lifecycle events derived from `contract_events`.

Key columns:
- `event_id` — matches the original contract event.
- `loan_id` — the domain loan identifier.
- `borrower` — loan borrower address.
- `event_type` — normalized loan event type.
- `amount` — loan principal, repayment amount, or other numeric detail.
- `status` — loan status or event-specific state.
- `occurred_at` — when the on-chain event happened.
- `created_at` — insertion timestamp.

Why it exists: makes loan-specific queries easier by flattening domain-relevant fields in a dedicated table.

### `scores`
Purpose: maintain the current credit score for each user.

Key columns:
- `user_id` — primary key for the score record.
- `score` — current numeric score value.
- `score_tier` — optional bucketed score classification.
- `loan_count` — number of loans used in scoring.
- `last_updated` — timestamp of last score change.

Why it exists: enables fast query of the current score without replaying events.

### `webhook_subscriptions`
Purpose: store webhook subscriber configuration.

Key columns:
- `subscription_id` — primary key.
- `target_url` — destination endpoint.
- `event_types` — list of event types to send.
- `is_active` — subscription status.
- `secret` — signing secret or verification token.
- `created_at` / `updated_at` — lifecycle timestamps.

Why it exists: records which external endpoints should receive event notifications.

### `webhook_deliveries`
Purpose: track webhook delivery attempts for each subscription and event.

Key columns:
- `delivery_id` — primary key.
- `subscription_id` — foreign key to `webhook_subscriptions`.
- `event_id` — foreign key to `contract_events` or `loan_events`.
- `status` — delivery outcome (`pending`, `sent`, `failed`).
- `attempt_count` — number of tries.
- `response_code` — HTTP status from the webhook target.
- `error_message` — failure details when delivery fails.
- `last_attempted_at` — timestamp of last attempt.

Why it exists: provides visibility into webhook reliability and retry behavior.

### `notifications`
Purpose: record notifications generated by the backend.

Key columns:
- `notification_id` — primary key.
- `user_id` — recipient identifier.
- `type` — notification category.
- `channel` — delivery channel (`email`, `sms`, etc.).
- `payload` — structured notification content.
- `status` — `queued`, `sent`, `failed`.
- `created_at` / `sent_at` — lifecycle timestamps.

Why it exists: separates notification intent from delivery state.

### `audit_logs`
Purpose: capture immutable operational history for important changes.

Key columns:
- `audit_id` — primary key.
- `entity_type` — table or domain entity changed.
- `entity_id` — identifier of the changed record.
- `action` — operation performed (`insert`, `update`, `delete`).
- `performed_by` — user or system actor.
- `changes` — JSON or text describing the change.
- `created_at` — timestamp.

Why it exists: supports debugging and postmortem analysis.

### `idempotency_keys`
Purpose: prevent duplicate processing of repeated requests.

Key columns:
- `idempotency_key` — primary key for the dedupe key.
- `request_hash` — optional fingerprint of the request payload.
- `response_status` — stored response status or result key.
- `created_at` — timestamp of first use.
- `expires_at` — key expiry to keep the table bounded.

Why it exists: ensures API and webhook operations remain idempotent.

## Important indexes

### `contract_events`
- `PRIMARY KEY (event_id)`
- `UNIQUE (ledger, transaction_hash, contract_id, cursor)` for deduplication and replay safety.
- Index on `(contract_id, event_type)` for event-type filtering.
- Index on `(ledger)` to support fast range queries by ledger range.
- Index on `(created_at)` for time-based audits.

### `loan_events`
- `PRIMARY KEY (event_id)`
- Index on `(loan_id)` for loan history queries.
- Index on `(borrower)` to retrieve borrower event streams.
- Index on `(event_type)` for event-specific processing.

### `scores`
- `PRIMARY KEY (user_id)`
- Index on `(score)` if sorted score queries are required.

### `webhook_subscriptions`
- `PRIMARY KEY (subscription_id)`
- Index on `(is_active)` for active subscriber lookup.
- `GIN` index on `event_types` if stored as an array or JSONB.

### `webhook_deliveries`
- `PRIMARY KEY (delivery_id)`
- Index on `(subscription_id, status)` for retry and status queries.
- Index on `(event_id)` to link deliveries back to events.

### `notifications`
- `PRIMARY KEY (notification_id)`
- Index on `(user_id, status)` for delivery queue processing.

### `audit_logs`
- `PRIMARY KEY (audit_id)`
- Index on `(entity_type, entity_id, created_at)` for audit retrieval.

### `idempotency_keys`
- `PRIMARY KEY (idempotency_key)`
- Index on `(expires_at)` for periodic cleanup.

## Unified `contract_events` / `loan_events` view

A shared view simplifies querying the event stream across raw and normalized tables. For example:

- `contract_events_history_view` combines `contract_events` with `loan_events`.
- It exposes common columns such as `event_id`, `contract_id`, `loan_id`, `event_type`, `borrower`, `amount`, `ledger`, `transaction_hash`, `occurred_at`, and `payload`.

Why it matters:
- Consumers can query one consistent event stream instead of joining multiple tables manually.
- It preserves the raw event history while allowing loan-specific fields to be surfaced in the same result set.

## How the schema maps to on-chain events via the indexer

The indexer is the glue between Soroban on-chain events and PostgreSQL:

- The indexer polls the Stellar RPC node for new contract events.
- Each decoded on-chain event is persisted first into `contract_events`.
- Events that represent loan lifecycle transitions are normalized into `loan_events`.
- The indexer also updates derived state such as `scores` and triggers webhook deliveries.
- `event_id`, `ledger`, `transaction_hash`, and `cursor` preserve the on-chain origin so audit and replay are possible.

Example mapping:
- `LoanRequested` -> `contract_events.event_type = 'LoanRequested'`, plus a `loan_events` row with `loan_id`, `borrower`, and `amount`.
- `LoanRepaid` -> `contract_events.event_type = 'LoanRepaid'` and a normalized row in `loan_events`.
- score-impacting events update `scores` based on the decoded event payload.
